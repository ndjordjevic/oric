Learn SystemVerilog · Lesson 0002

Wires, bit-slices, ternary & module instantiation

Source: core/Oric.sv, lines 48–61 (★ OSD MENU & ASPECT RATIO). Prior lesson: 0001 — modules, ports & assign.

The win for today

You'll be able to read a wire declaration with a width, pull a sub-range out of a wide bus, follow a ternary expression, and read a module instantiation that mixes auto-wiring (.*) with explicit named ports.

1. wire [1:0] ar = status[122:121]; — declaring and wiring in one line

49wire [1:0] ar = status[122:121];

wire [1:0] ar declares a 2-bit-wide signal named ar. The [1:0] means "bit 1 down to bit 0" — two bits total, indexed high-to-low. The = status[122:121] immediately wires it to a 2-bit slice pulled out of status, a much wider (128-bit) bus declared elsewhere.

A wire declared with an initializer is like const ar = status.slice(121, 123) in JS, except it's not a one-time copy — like assign, it's a live formula: whenever status changes, ar updates instantly.
[msb:lsb] range syntax is always high bit first. status[122:121] means bits 122 and 121 — NOT a Python-style slice where the second number is exclusive. Both ends are inclusive, and the width is 122-121+1 = 2 bits, matching ar's declared width.

2. The ternary operator ?: — the same one you already know

56.ARX((!ar) ? 12'd4 : (ar - 1'd1)),
57.ARY((!ar) ? 12'd3 : 12'd0),

SystemVerilog's cond ? a : b is exactly the ternary operator from C/JS/Python's a if cond else b. If ar is zero (!ar is true), ARX gets the sized literal 12'd4; otherwise it gets ar - 1.

Directly the same as JavaScript: const ARX = (!ar) ? 4 : (ar - 1); — no new concept here, just new surrounding syntax (sized literals, ports).

Sized literals: 12'd4

12'd4 reads as "a 4, forced into 12 bits, in decimal." The general form is <width>'<base><value> where base is b (binary), d (decimal), h (hex), or o (octal). You saw 2'd0, 8'h00, 1'b0 elsewhere in this file — same pattern, different width/base.

Why bother sizing a literal at all? Because in hardware, every wire has a fixed width, and an unsized literal's width is ambiguous in some contexts (it can silently zero- or sign-extend differently than you expect). Explicit sizing removes that ambiguity — treat it as documentation plus a safety net, not decoration.

3. Module instantiation: .* vs. named ports

50video_freak video_freak
51(
52	.*,
53	.VGA_DE_IN(VGA_DE),
54	.VGA_DE(),
56	.ARX((!ar) ? 12'd4 : (ar - 1'd1)),
57	.ARY((!ar) ? 12'd3 : 12'd0),
58	.CROP_SIZE(0),
59	.CROP_OFF(0),
60	.SCALE(status[16:15])
61);

video_freak video_freak ( ... ); instantiates one copy of the video_freak module, naming this particular copy video_freak too (module name and instance name are independent — reusing the same word is just a convention here).

A module instantiation is like constructing an object: const video_freak = new VideoFreak({ ARX: ..., ARY: ..., SCALE: status.slice(...) }). Each .PORT_NAME(value) is one constructor argument, matched by name rather than position.

.* is a shortcut meaning "auto-connect every port whose name matches a same-named signal already in scope" — it saves typing .CLK_VIDEO(CLK_VIDEO), .VGA_DE(VGA_DE), ... for every port that happens to share its name with an existing wire. Anything listed explicitly afterward (like .VGA_DE_IN(VGA_DE), connecting a differently-named port) overrides the auto-connect for that one port.

.VGA_DE() — empty parentheses — means "leave this output port unconnected." It's valid and common for outputs you don't care about, but it would be meaningless for a required input.

Quick check

Q1. How many bits wide is status[16:15], and which port receives it above?
Answer: 2 bits (15 and 16 inclusive); it's wired to the .SCALE port of video_freak.
Q2. What does .* do that plain named ports don't?
Answer: it auto-connects any port whose name exactly matches an already-declared signal in the surrounding scope, without writing out .name(name) for each one.
Q3. In 12'd4, what do the 12 and the d each mean?
Answer: 12 is the bit-width the literal is forced into; d says the number that follows is written in decimal.
You now have this in hand: sized wire declarations, [msb:lsb] bit-slicing, the ternary operator, sized literals, and module instantiation via both .* auto-wiring and explicit .PORT(signal) named connections — including leaving a port deliberately unconnected.
