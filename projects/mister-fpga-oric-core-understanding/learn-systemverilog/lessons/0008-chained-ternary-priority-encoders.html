Learn SystemVerilog · Lesson 0008

Chained ternary as a priority encoder

Source: annotated/Oric.sv, lines 498–508 (★ ROM SELECTION & FDD READY) and lines 616–622 (file-cache address mux). Prior: Lesson 7.

The win for today

You'll read a chain of stacked ternary operators as what it really is: an ordered list of "if this, else if that" cases collapsed into one expression — the same pattern shows up at least twice more later in the file.

1. Chaining ?: like an if/else-if ladder

500wire [1:0] rom_sel =
501	(rom == 2'd0) ? 2'd2 :
502	(rom == 2'd1) ? 2'd0 :
503	(rom == 2'd2) ? 2'd1 :
504	(bios_loaded ? 2'd3 : 2'd0);

Each cond ? a : b can nest inside another's "else" branch, producing a top-to-bottom priority chain: check the first condition; if false, its : ... branch is itself another ternary, checked next; and so on. Read line-by-line, top to bottom, exactly like a priority list:

This is precisely an if/else-if/else ladder written as one expression — the JS equivalent would be: 
const romSel =
  (rom === 0) ? 2 :
  (rom === 1) ? 0 :
  (rom === 2) ? 1 :
  (biosLoaded ? 3 : 0);
Same top-to-bottom priority reading, same "first match wins" semantics — just written as nested ternaries instead of an if chain, because this whole thing has to be one continuously-live wire expression, not a procedural statement.
This particular chain is a lookup-table remap: the menu's ROM option order (0,1,2,3) doesn't match the internal ROM numbering, so this translates one numbering into the other. The logic itself isn't complicated once you read it as a priority list — the only trap is trying to parse all four lines "at once" instead of top-to-bottom.

2. The same pattern used for muxing an address, not just a constant

616wire [FILE_CACHE_ADDR_WIDTH-1:0] file_selected_addr =
617  file_download_active ? file_download_addr :
618  ss_load_active       ? ss_fc_addr :
619  snap_active          ? snap_cache_addr :
620  tap_active           ? tap_cache_addr :
621  tap_byte_active      ? tap_byte_cache_addr :
622                         tape_addr;

Identical shape to the ROM-select chain, just picking between five different address buses instead of five constants — whichever subsystem is currently "active" gets to drive the shared file-cache address, in a fixed priority order (download > save-state load > snapshot restore > tape segment loader > tape byte streamer > default cassette playback).

Notice there's no explicit final condition on line 622 — tape_addr is the unconditional fallback if none of the five _active flags are set, exactly like a final bare else at the end of an if/else-if ladder.

Quick check

Q1. If rom == 2'd1 is true, what value does rom_sel get, regardless of whether later conditions would also be true?
Answer: 2'd0 — the first matching condition in a ternary chain wins; later conditions are never even evaluated for their effect once an earlier one matches.
Q2. In the second example, what drives file_selected_addr when none of the five named "_active" signals are true?
Answer: tape_addr — the final unconditional fallback value at the end of the chain.
You now have this in hand: reading a nested/stacked ternary chain as an ordered priority list — top-to-bottom, first match wins, with an optional unconditional fallback at the end. This single idiom recurs throughout the codebase wherever several subsystems compete to drive one shared signal.
