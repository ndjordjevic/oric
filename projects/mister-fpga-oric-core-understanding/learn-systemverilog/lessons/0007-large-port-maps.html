Learn SystemVerilog · Lesson 0007

Reading a large port map

Source: annotated/Oric.sv, lines 389–493 (★ ORIC ATMOS COMPUTER CORE — the single biggest instantiation in the file). Prior: Lesson 6.

The win for today

No new syntax appears here — the point of this lesson is fluency: reading a 100-line port map without losing your place, by recognizing the same small set of patterns from Lessons 1–6 repeated at scale.

1. It's the same shape as every instantiation you've already read

389oricatmos oricatmos
390(
391	.clk_in           (clk_sys),
392	.RESET            (reset),
393	.key_pressed      (kbd_ps2_key[9]),
394	.key_code         (kbd_ps2_key[7:0]),
395	.key_extended     (kbd_ps2_key[8]),

module_name instance_name ( .port(signal), ... ) — identical shape to video_freak (Lesson 2), pll and hps_io (Lesson 5). The only thing that's changed is scale: oricatmos (defined in rtl/oricatmos.vhd — see 01b-oricatmos-vhd-understanding.md) has roughly 90 ports, because it is the whole computer — CPU, video, sound, keyboard, tape, floppy all in one module.

Line 391 wires the port clk_in to the signal clk_sys — different names either side of the parens is completely normal and expected; the port name belongs to oricatmos's internal namespace, the signal name belongs to Oric.sv's. Don't assume matching names means anything special, and don't assume different names mean anything special either — always check which side of .port(signal) you're looking at.

2. Bit-slices inline in a port connection — nothing new, just combined

393.key_pressed      (kbd_ps2_key[9]),
394.key_code         (kbd_ps2_key[7:0]),
395.key_extended     (kbd_ps2_key[8]),

One 11-bit bus (kbd_ps2_key) is being split across three different ports: bit 9 alone, bits 7 down to 0 as an 8-bit byte, and bit 8 alone. This is exactly the [msb:lsb] slicing from Lesson 2 — it just happens to sit directly inside a port connection instead of on the right of a wire declaration.

3. A ternary expression inline in a port connection

428.disk_enable      ((!status[6:5]) ? ~fdd_ready : status[5]),

Again, nothing new: the ternary operator from Lesson 2, just written directly as a port's value instead of first being assigned to a named wire. Anywhere a signal is expected, a full expression is allowed — SystemVerilog doesn't require you to pre-name every intermediate value.

Same idea as inlining an expression as a function argument in JS: configure({ diskEnable: !status ? !fddReady : status }) instead of computing a named variable first. Both are legal; the codebase mixes styles depending on whether the expression is reused elsewhere (named wire) or used exactly once (inlined).

4. Comments inside a port map, and an unconnected output

416.ram_oe           (),
426.phi2             (),
472	// mode write-back is shared by the loader (after restore) and the
473	// save engine (after the RAM dump) — never active simultaneously
474	.ula_snap_mode_we (ula_snap_mode_we | ss_ula_mode_we),

() again means "this output is deliberately left unconnected" (Lesson 5) — ram_oe and phi2 are outputs Oric.sv simply doesn't need. Regular // line comments are allowed anywhere, including inside a port list, and are worth reading — they often explain a non-obvious sharing arrangement between two signals feeding the same port (line 474 combines two write-enable sources with bitwise-OR, and the comment explains why that's safe: they're never both active at once).

Quick check

Q1. On line 394, what does kbd_ps2_key[7:0] represent, and how wide is it?
Answer: the low 8 bits of the 11-bit kbd_ps2_key bus — the ASCII/scan-code byte portion of the key event, 8 bits wide.
Q2. Is there any rule preventing an expression (like a ternary) from being written directly inside a port connection, instead of via a named wire first?
Answer: no — any valid expression can appear directly as a port's connected value; naming it first via a wire is a style choice for readability/reuse, not a language requirement.
You now have this in hand: confidence reading a large, real port map at a glance — recognizing it as the same handful of patterns (named connection, bit-slice, ternary, unconnected output, inline comment) repeated, not new syntax to fear.
