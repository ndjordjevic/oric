Learn SystemVerilog · Lesson 0009

Local variables, replication & clock-domain crossing

Source: annotated/Oric.sv, lines 512–546 (★ VIDEO OUTPUT PIPELINE). Prior: Lesson 8.

The win for today

You'll read a variable declared inside an always block, understand the double-flop pattern used whenever a signal crosses from one clock to another, and read the replication operator {n{x}}.

1. A reg declared inside an always block

516reg ce_pix;
517always @(posedge CLK_VIDEO) begin
518	reg old_clk;
519
520	old_clk <= clk_pix2;
521	ce_pix <= ~old_clk & clk_pix2;
522end

Line 518 declares old_clk inside the always block, right after begin. This is legal, and — because it's never referenced outside this block — it behaves as a private register belonging only to this one clocked process, still with memory across cycles (unlike a plain local variable in software that resets every function call).

The closest software analogy is a variable declared with static inside a C function, or a closure variable in JS that persists between calls — declared in a narrow scope, but its value survives from one invocation (clock edge) to the next.
Two different clocks appear here: this whole block runs on posedge CLK_VIDEO (the HDMI pixel clock), while clk_pix2 was itself set by an earlier, separate always @(posedge clk_sys) block (the system clock) just above it in the file. That's not a mistake — it's the clock-domain-crossing pattern below.

2. The double-flop synchronizer: crossing between clocks safely

513reg clk_pix2;
514always @(posedge clk_sys) clk_pix2 <= clk_pix;

Here's the full picture: clk_pix is generated inside the oricatmos core, ticking on its own schedule. To safely read it from the CLK_VIDEO domain without risking a signal caught mid-transition (a real hardware hazard when two independent clocks interact), the design re-samples it twice: first into clk_pix2 (on clk_sys), then into old_clk (on CLK_VIDEO, line 520). ce_pix then pulses for one CLK_VIDEO cycle exactly when a rising edge of the (already-resynchronized) pixel clock is detected — the same edge-detection idiom from Lesson 6, just combining a bitwise-AND (&, two operands here — not reduction) with a NOT.

There's no clean single-clock software analogy — this is a genuinely hardware-specific safety pattern, needed because two clocks in a real chip are physically independent oscillators that can tick at any relative phase. The closest conceptual parallel is debouncing a physical button: you don't trust the very first raw reading, you let it settle across a couple of samples first.

You don't need to master clock-domain-crossing theory to keep reading this file — just recognize the shape: a signal sampled by one clock, then re-sampled again by a second clock, before anything in the second domain is allowed to act on it.

3. The replication operator: {n{x}}

542.R({4{r}}),
543.G({4{g}}),
544.B({4{b}}),

{4{r}} repeats the single bit r four times, producing a 4-bit bus where every bit equals r. The Oric only has 1-bit-per-channel video (fully on or fully off), but HDMI wants 4 bits per channel — replication is the cheap way to expand a 1-bit signal to a wider bus without changing its meaning (all-1s if on, all-0s if off).

For placement: these three lines sit inside the video_mixer instantiation (lines 539–546) — .R, .G, .B are its per-channel color input ports, and r, g, b are the core's 1-bit red, green, and blue video signals.

Equivalent to JS's Array(4).fill(r).join('') if you imagine bits as characters — repeat the same value n times to build a wider value. It's concatenation's close cousin: {a, b} glues different values together; {n{x}} glues n copies of the same value together.

Quick check

Q1. Why does the pixel clock get sampled twice (into clk_pix2, then old_clk) instead of being used directly by CLK_VIDEO's logic?
Answer: because clk_pix comes from a different, independent clock domain — reading it directly risks catching it mid-transition; sampling it through two flip-flops in sequence lets it settle safely before anything acts on it.
Q2. What's the resulting width and value of {4{1'b1}}?
Answer: a 4-bit value, all four bits equal to 1 — i.e. 4'b1111.
You now have this in hand: block-scoped reg declarations inside always, the double-flop clock-domain-crossing shape (recognize it, don't need to design one yet), and the replication operator {n{x}} for widening a narrow signal.
