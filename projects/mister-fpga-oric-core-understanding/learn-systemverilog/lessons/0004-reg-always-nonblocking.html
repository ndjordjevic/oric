Learn SystemVerilog · Lesson 0004

reg, always @(posedge) & non-blocking assignment

Source: annotated/Oric.sv, lines 150–173 (★ RESET & RAM CLEAR). Prior: Lesson 3. This is the most important lesson so far — it's the syntax of sequential logic, i.e. anything with memory.

The win for today

You'll be able to read a clocked block that counts, resets, and toggles — the pattern every flip-flop, counter, and state machine in this codebase is built from.

1. reg vs wire: who's allowed to hold a value

151reg        reset = 0;
152reg [16:0] clr_addr = 0;

wire (Lessons 1–3) can only ever be a live formula — it must be continuously driven by an assign or a port connection, and has no memory of its own. reg is different: it's a signal that can be written inside a procedural block (an always block, coming up next) and holds its value between writes — it's memory.

Think of wire as a computed property (a getter, always re-derived) and reg as a plain mutable variable that only changes when explicit code assigns to it. The = 0 here is the reset/power-on initial value — for FPGA synthesis this becomes the value the flip-flop powers up with, not a "default if unwritten" like in software.

2. always @(posedge clk_sys) begin ... end — the heartbeat block

155always @(posedge clk_sys) begin
156
157	if(~&clr_addr) clr_addr <= clr_addr + 1'd1;
158	else reset <= 0;
159
160	if(manual_reset_req | tap_autorun_reset_req) begin
161		clr_addr <= 0;
162		reset <= 1;
163	end
165end

always @(posedge clk_sys) means "run everything inside this block exactly once, at the instant clk_sys transitions from 0 to 1" (a rising edge) — repeated forever, once per clock cycle. begin ... end groups multiple statements into one block, exactly like curly braces { } group statements in C/JS (SystemVerilog reserves { } for concatenation, so it needs different block delimiters). And just like braces in C/JS, begin/end is optional when the body is a single statement — later in the file you'll see one-liners like always @(posedge clk_sys) if(reset) rom <= status[4:3]; with no begin/end at all.

The closest software analogy is an event handler that fires once per clock tick: clock.on('risingEdge', () => { ... }). But unlike a JS event handler, everything inside conceptually happens simultaneously and instantaneously at that clock edge — there's no "first this line runs, then that one takes effect" the way software statements execute in sequence.

3. <= — non-blocking assignment (the crucial hardware-only idea)

Inside an always block, <= means: "at the next clock edge, this signal will become this value" — all the right-hand sides in the block are read using the old values, and all the updates land together, atomically, as one snapshot.

This is the single most important, least intuitive rule in this whole lesson series. Look at lines 160–163: clr_addr <= 0; and reset <= 1; both use the values as they were before this clock edge to decide what to become — they do NOT see each other's new values within the same block execution. It's as if every <= in the block were computed first from old values, then all applied at once. This models real flip-flops: their outputs only change at the clock edge, and every flip-flop reads its input from before that edge.
If you've used React's setState, this is a decent analogy: calling setState doesn't change the value immediately — it queues an update that's applied together with other updates, using the state as it was at the start of the current render. <= behaves the same way inside one clock edge.

4. Priority via if / else if, and two reduction operators

157if(~&clr_addr) clr_addr <= clr_addr + 1'd1;
158else reset <= 0;

&clr_addr is the reduction-AND operator: it ANDs together every single bit of clr_addr into one result — true only if all bits are 1 (i.e. the counter has reached its maximum value, all-1s, meaning RAM-clearing is done). ~&clr_addr then flips that: true while the counter has not yet reached all-1s.

There's no single-operator equivalent in software, but the effect is exactly clr_addr !== MAX_VALUE in JS — "not yet finished counting." SystemVerilog spells that as a bit-reduction instead of a comparison, but the intent is identical.

if (cond) stmt1; else stmt2; reads exactly like C/JS/Python's if/else. Because this is inside a clocked always block, "else" here doesn't mean "otherwise, right now" — it means "otherwise, at the next clock edge."

One more rule this block relies on: if the same signal is given a non-blocking assignment more than once inside one always block, only the last one written in the code actually takes effect at the next edge — later statements silently override earlier ones for that signal, even though both "happened." Look at lines 157–163: the first if/else (157–158) assigns clr_addr and reset based on the counter; the second, separate if (160–163) can assign the very same two signals again, unconditionally, if a reset is requested. When both fire on the same edge, the second block's assignments are what the flip-flops actually latch — not because it "runs after" in time (everything here happens at one clock edge), but simply because it's the last assignment to each signal in the block's source-code order.

5. A second block, same shape: toggling a clock divider

167wire tape_clk;
168always @(posedge clk_sys) begin
169	if (reset)
170    	tape_clk <= 1'b0;
171	else
172    	tape_clk <= ~tape_clk;
173end
Wait — tape_clk is declared wire above but assigned with <= inside an always block? That's actually a latent bug/inconsistency in the real source (it should be reg) — some tools accept it, but it's not something to imitate. Worth noting precisely because spotting "that doesn't look right" is a sign you're actually reading the syntax rules, not just pattern-matching.

Ignoring that wrinkle, the logic itself is a classic clock divider: every cycle, if not resetting, tape_clk flips to its own opposite (~tape_clk) — producing a signal that toggles once per clk_sys cycle, i.e. runs at half the frequency.

Quick check

Q1. In the block on lines 155–165, if both the counter-increment condition and the manual-reset condition were true on the same clock edge, which one wins, and why?
Answer: the reset block (lines 160–163) wins, because it appears later in the same always block and unconditionally overwrites both clr_addr and reset after the earlier if/else already queued its own update — later non-blocking assignments to the same signal in one block simply override earlier ones.
Q2. Why can't &clr_addr just be written as clr_addr == 17'h1FFFF and mean something different?
Answer: they mean the same thing here (both true only when every bit is 1) — reduction-AND is just a shorter, width-independent way to express "all bits set" without hardcoding the all-ones constant for a specific width.
Q3. What real hardware component does one always @(posedge clk) block with <= assignments model?
Answer: a flip-flop (or a bank of flip-flops) — a memory element that samples its input and updates its output only at the clock's rising edge.
You now have this in hand: reg as stateful memory, always @(posedge clk) as the clocked-block trigger, begin/end as SystemVerilog's block delimiter, non-blocking <= and why it differs fundamentally from software assignment, reduction-AND &, and if/else priority inside a clocked block. This unlocks reading almost every stateful signal in the file.
