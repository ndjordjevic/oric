Modules, ports & continuous assignment

Source: annotated/Oric.sv, lines 22–46 (the ★ MODULE SETUP section) — same lines summarized functionally in 01a-Oric-sv-understanding.md §1. This lesson covers the syntax only. All lessons cite line numbers from annotated/Oric.sv (the frozen study copy with ★ section comments); the pristine core/Oric.sv is the same code minus those comments, so its line numbers sit slightly lower.

The win for today

By the end of this lesson you'll be able to read the opening lines of any SystemVerilog file — the module/port skeleton and a block of assign statements — and know exactly what's being declared versus what's happening every clock cycle.

1. A module is a component, not a function call

 22module emu
 23(
 24	`include "sys/emu_ports.vh"
 25);
 26
 27// ★ MODULE SETUP...
 28assign USER_OUT = '1;

module emu ( ... ); declares a hardware block named emu with a fixed set of pins listed between the parentheses. Everything from this line to the matching endmodule (at the very end of the file) is that block's internals.

Think of module like a class declaration in Python or a struct with fixed fields in C — it defines a shape. But unlike a function, nothing inside it "runs in order" from top to bottom. Every statement inside describes a piece of wiring or a piece of behavior that exists all the time, simultaneously — more like a spreadsheet of formulas than a script.

Line 24 is unusual: instead of listing ports directly, it uses a preprocessor directive, `include "sys/emu_ports.vh", to paste in a giant shared port list from another file. This is the Verilog preprocessor (same family of tool as C's #include) — it happens before the compiler ever sees the actual module, purely textual substitution.

The backtick before include is not a typo. SystemVerilog preprocessor directives (`include, `define, `ifdef) all start with a backtick, to distinguish them from ordinary code. If you see a leading backtick, mentally read it as "textual copy-paste happens here before compilation," exactly like C's #include.

2. assign: a permanently-live connection, not a one-time write

 28assign USER_OUT = '1;
 29assign {UART_RTS, UART_TXD, UART_DTR} = 0;
 33assign DDRAM_CLK = clk_sys;
 35assign LED_USER    = ioctl_download | fdd_busy | tape_adc_act | led_user_pokeable;

assign left = right; is a continuous assignment: it wires the right-hand expression directly to the left-hand signal, permanently. Whenever any input on the right changes, the output on the left updates immediately — there is no "moment" it runs, because it never stops running.

In JavaScript/React terms, this is closer to a computed/derived value (like a spreadsheet cell =A1+B1) than to a variable assignment x = 5. It's not "set x to 5 right now" — it's "x is now permanently defined as whatever the formula evaluates to." If A1 changes, the cell updates on its own.

Line 35 shows this doing real work: LED_USER lights whenever any of four conditions is true, via the bitwise-OR operator |. Because it's a continuous assignment, the LED reacts instantly and automatically as any of those four signals changes — no one has to "call" this logic.

3. Curly braces { } mean concatenation, not a block or an object

 29assign {UART_RTS, UART_TXD, UART_DTR} = 0;
 31assign {SDRAM_DQ, SDRAM_A, SDRAM_BA, SDRAM_CLK, SDRAM_CKE, SDRAM_DQML, SDRAM_DQMH, SDRAM_nWE, SDRAM_nCAS, SDRAM_nRAS, SDRAM_nCS} = 'Z;

In SystemVerilog, { } is the concatenation operator — it glues several signals together end-to-end into one wider bus, or (as here, on the left of an assign) splits one value out across several signals at once.

This is like Python/JS destructuring assignment: [UART_RTS, UART_TXD, UART_DTR] = [0, 0, 0] — one value (0) fanned out to every named target in the list, all at once, forever (because it's an assign, not a one-time statement).
Don't confuse SystemVerilog's { } with C/JS curly braces for code blocks or object literals — SystemVerilog uses begin ... end for blocks (not seen in this lesson's snippet) and has no object-literal syntax at all here. { } in an expression is always bit-concatenation.

4. Number literals: '1, 'Z, and plain 0

 28assign USER_OUT = '1;
 31assign {...} = 'Z;
 37assign LED_POWER   = 0;
LiteralMeaning
'1fill every bit of the target with 1 (width taken from the left-hand signal)
'Zfill every bit with high-impedance — electrically disconnected, not driving 0 or 1
0ordinary numeric literal, zero-extended to fit the target width
There's no clean software analogy for 'Z — it's a genuinely hardware-only idea. The closest mental model: imagine a shared bus where multiple devices could try to drive it, like several people who could each shout on a party line. 'Z means "I'm hanging up — not saying anything, not even silence, just off the line" so someone else can drive that wire without a conflict.

Line 31 sets a whole bundle of SDRAM pins to 'Z because this core doesn't use external SDRAM — it releases those pins entirely rather than driving them to a meaningless 0.

Quick check

Q1. What does assign x = a | b; do the instant a changes from 0 to 1, assuming b stays 0?
Answer: x becomes 1 immediately — the assignment is continuously live, so it re-evaluates the moment any input changes, with no delay or trigger needed.
Q2. Rewrite assign {p, q} = 2'b10; in plain English — what value does each of p and q get?
Answer: p gets 1 (the leftmost/high bit of 10) and q gets 0 (the rightmost/low bit) — concatenation on the left splits the right-hand value bit-by-bit across the listed targets, left to right.
Q3. Why is 'Z not a meaningful value to test inside an if condition the way 0 or 1 are?
Answer: 'Z means the wire isn't being driven at all — it has no defined logic value, so a condition treats it as false/unknown rather than as real data; it's a statement about the wire's electrical state, not a value you branch on.
You now have this in hand: module/port skeletons, `include as textual paste, assign as a permanently-live formula (not a one-time write), { } as concatenation/destructuring, and the three literal forms '1 / 'Z / plain numbers. That's enough to read all of Oric.sv's ★ MODULE SETUP section unaided.
