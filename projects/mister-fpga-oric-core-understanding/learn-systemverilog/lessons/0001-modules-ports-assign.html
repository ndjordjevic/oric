Modules, ports & continuous assignment

Source: annotated/Oric.sv, lines 22–46 (the ★ MODULE SETUP section) — same lines summarized functionally in 01a-Oric-sv-understanding.md §1. This lesson covers the syntax only. All lessons cite line numbers from annotated/Oric.sv (the frozen study copy with ★ section comments); the pristine core/Oric.sv is the same code minus those comments, so its line numbers sit slightly lower.

The win for today

By the end of this lesson you'll be able to read the opening lines of any SystemVerilog file — the module ... endmodule skeleton with its port list, and a block of assign statements — and know exactly what's being declared versus what's happening every clock cycle. (Note: module is a literal keyword you'll see in the code; "port" is just the concept — SystemVerilog has no port keyword.)

1. A module is a component, not a function call

 22module emu
 23(
 24	`include "sys/emu_ports.vh"
 25);
 26
 27// ★ MODULE SETUP...
 28assign USER_OUT = '1;

module emu ( ... ); declares a hardware block named emu with a fixed set of pins listed between the parentheses. Everything from this line to the matching endmodule (at the very end of the file) is that block's internals.

Think of module like a class declaration in Python or a struct with fixed fields in C — it defines a shape. But unlike a function, nothing inside it "runs in order" from top to bottom. Every statement inside describes a piece of wiring or a piece of behavior that exists all the time, simultaneously — more like a spreadsheet of formulas than a script.

Line 24 is unusual: instead of listing ports directly, it uses a preprocessor directive, `include "sys/emu_ports.vh", to paste in a giant shared port list from another file. This is the Verilog preprocessor (same family of tool as C's #include) — it happens before the compiler ever sees the actual module, purely textual substitution.

The backtick before include is not a typo. SystemVerilog preprocessor directives (`include, `define, `ifdef) all start with a backtick, to distinguish them from ordinary code. If you see a leading backtick, mentally read it as "textual copy-paste happens here before compilation," exactly like C's #include.

2. assign: a permanently-live connection, not a one-time write

 28assign USER_OUT = '1;
 29assign {UART_RTS, UART_TXD, UART_DTR} = 0;
 33assign DDRAM_CLK = clk_sys;
 35assign LED_USER    = ioctl_download | fdd_busy | tape_adc_act | led_user_pokeable;
 36assign LED_DISK    = led_disk;

assign left = right; is a continuous assignment: it wires the right-hand expression directly to the left-hand signal, permanently. Whenever any input on the right changes, the output on the left updates immediately — there is no "moment" it runs, because it never stops running.

In JavaScript/React terms, this is closer to a computed/derived value (like a spreadsheet cell =A1+B1) than to a variable assignment x = 5. It's not "set x to 5 right now" — it's "x is now permanently defined as whatever the formula evaluates to." If A1 changes, the cell updates on its own.

Line 35 shows this doing real work: LED_USER lights whenever any of four conditions is true, via the bitwise-OR operator |. Because it's a continuous assignment, the LED reacts instantly and automatically as any of those four signals changes — no one has to "call" this logic. Line 36 is the simplest possible case of the same idea: LED_DISK is just wired straight through to whatever led_disk is doing, with no logic in between — a continuous assignment can be a bare wire-to-wire connection just as easily as a whole expression.

The other three lines of this excerpt get their right-hand sides explained in the next two sections, but their targets are worth naming now: USER_OUT (line 28) is the MiSTer USER I/O port's pin bundle, driven all-high because this core doesn't use it; line 29 ties off the three serial-port outputs UART_RTS, UART_TXD, UART_DTR — this core has no UART, so all three sit at 0; and line 33 wires DDRAM_CLK, the clock output of the DDR3 memory interface, straight to clk_sys — the master system clock, produced by a PLL further down the file (Lesson 5).

Notice ioctl_download, fdd_busy, tape_adc_act, led_user_pokeable, and led_disk are all used here on lines 35–36 but not declared anywhere yet — their wire/reg declarations don't appear until lines 203, 387, 914, 727, and 386 respectively, much later in the file. (led_disk is declared just one line before fdd_busy, at line 386 — both come from the floppy-disk-controller section, which is why they sit next to each other.) In a sequential language like C or Python, using a name before it's defined is an error. In SystemVerilog it's completely normal: since everything in a module happens concurrently rather than top-to-bottom (the "spreadsheet, not a script" idea from earlier), a module's signals don't need to be declared before the point where they're first used — the compiler sees the whole file before anything "runs." Don't expect the meaning of an unfamiliar name to be explained where you first see it used; it may be declared (and its purpose explained) hundreds of lines later.

3. Curly braces { } mean concatenation, not a block or an object

 29assign {UART_RTS, UART_TXD, UART_DTR} = 0;
 31assign {SDRAM_DQ, SDRAM_A, SDRAM_BA, SDRAM_CLK, SDRAM_CKE, SDRAM_DQML, SDRAM_DQMH, SDRAM_nWE, SDRAM_nCAS, SDRAM_nRAS, SDRAM_nCS} = 'Z;

In SystemVerilog, { } is the concatenation operator — it glues several signals together end-to-end into one wider bus, or (as here, on the left of an assign) splits one value out across several signals at once.

This is like Python/JS destructuring assignment: [UART_RTS, UART_TXD, UART_DTR] = [0, 0, 0] — one value (0) fanned out to every named target in the list, all at once, forever (because it's an assign, not a one-time statement).
Don't confuse SystemVerilog's { } with C/JS curly braces for code blocks or object literals — SystemVerilog uses begin ... end for blocks (not seen in this lesson's snippet) and has no object-literal syntax at all here. { } in an expression is always bit-concatenation.

4. Number literals: '1, 'Z, and plain 0

 28assign USER_OUT = '1;
 31assign {...} = 'Z;
 37assign LED_POWER   = 0;
LiteralMeaning
'1fill every bit of the target with 1 (width taken from the left-hand signal)
'Zfill every bit with high-impedance — electrically disconnected, not driving 0 or 1
0ordinary numeric literal, zero-extended to fit the target width
There's no clean software analogy for 'Z — it's a genuinely hardware-only idea. The closest mental model: imagine a shared bus where multiple devices could try to drive it, like several people who could each shout on a party line. 'Z means "I'm hanging up — not saying anything, not even silence, just off the line" so someone else can drive that wire without a conflict.

Line 30 does the same "release the pins" trick as line 31, just for a different peripheral: assign {SD_SCK, SD_MOSI, SD_CS} = 'Z; puts the SD-card SPI pins into high-impedance because this particular build of the core doesn't drive an SD card itself. Line 31 sets a whole bundle of SDRAM pins to 'Z because this core doesn't use external SDRAM — it releases those pins entirely rather than driving them to a meaningless 0. The comment directly above line 33 (// DDRAM is used as the MiSTer Main savestate buffer) is an ordinary // line comment — plain English, ignored by the compiler — explaining that unlike SD/SDRAM, DDRAM_CLK on line 33 is not tied off, because DDRAM is actually in use elsewhere in the file for save states.

The plain 0 on line 37 is the odd one out of the three, and deliberately so: it's not special hardware syntax at all — it's the same ordinary decimal literal you already know from every other language. The only SystemVerilog-specific behavior is what happens to it when it's assigned to a wider target: assign LED_POWER = 0; zero-extends it to fill whatever width LED_POWER actually is. '1 and 'Z exist specifically because "ordinary" literals like 0 can't express "every bit set" or "electrically disconnected" without knowing the target's width in advance — they're the two special-purpose forms; plain 0 is the baseline everything else is contrasted against.

5. The rest of the tie-off block (lines 38–46): same pattern, different pins

 38assign BUTTONS     = 0;
 39assign VGA_SCALER  = 0;
 40assign VGA_DISABLE = 0;
 41assign HDMI_FREEZE = 0;
 42assign HDMI_BLACKOUT = 0;
 43assign HDMI_BOB_DEINT = 0;
 45assign AUDIO_S   = 0;
 46assign AUDIO_MIX = 0;

Every one of these eight lines uses the exact same construct already covered above: a single-signal continuous assignment to the plain literal 0, zero-extended to whatever width the target needs. There's no new syntax here — the reason to look at them individually is to know what each one means, since a decoder needs both the syntax and the intent:

  • BUTTONS = 0 — per core/sys/emu_ports.vh's own comment, this 2-bit output reports b[1]: user button and b[0]: osd button state back to the framework; tying both bits to 0 means this core never asserts either button itself.
  • VGA_SCALER = 0 — per the same file, this output would force the framework's VGA scaler on if asserted; tying it to 0 leaves that choice to the framework's own default instead of forcing it. VGA_DISABLE = 0 reports whether analog VGA output is off; 0 means it stays enabled.
  • HDMI_FREEZE = 0, HDMI_BLACKOUT = 0, HDMI_BOB_DEINT = 0 — three more optional MiSTer HDMI-framework output flags (undocumented by inline comment in emu_ports.vh, but following the same "core never needs to assert this" pattern as the rest of the block) permanently held low.
  • AUDIO_S = 0 — tells the framework this core's audio output is not signed (0 = unsigned format) rather than being an enable/disable flag.
  • AUDIO_MIX = 0 — selects the "no extra mixing" mode (0 = off) out of several audio-mixing options the framework supports.
Don't assume every = 0 in this style of block means "disabled" or "off" — as with AUDIO_S above, sometimes 0 selects a specific mode (unsigned audio) rather than switching a feature off. The syntax (assign x = 0;) never tells you the intent by itself; only the target signal's name and its purpose elsewhere in the framework do. This is exactly why this lesson series is scoped to syntax, not function — you'll need 01a-Oric-sv-understanding.md or the MiSTer framework's own port documentation to know what each specific tied-off signal actually controls, even once you fully understand the assign syntax itself.

Quick check

Q1. What does assign x = a | b; do the instant a changes from 0 to 1, assuming b stays 0?
Answer: x becomes 1 immediately — the assignment is continuously live, so it re-evaluates the moment any input changes, with no delay or trigger needed.
Q2. In assign {UART_RTS, UART_TXD, UART_DTR} = 0;, what value does each of the three signals get, and why don't they get different bits of 0?
Answer: all three get 0 — the right-hand side here is a single scalar value, not a multi-bit value being split apart, so concatenation on the left just broadcasts that one value to every listed target equally.
Q3. What does it mean, physically, for a pin to be driven to 'Z instead of 0?
Answer: the pin is electrically disconnected — not driving any voltage at all — rather than actively driving low; this matters when a wire might be shared with another device that needs to drive it instead.
You now have this in hand: the module ... endmodule skeleton and its port list, `include as textual paste, assign as a permanently-live formula (not a one-time write), { } as concatenation/destructuring, and the three literal forms '1 / 'Z / plain numbers. That's enough to read all of Oric.sv's ★ MODULE SETUP section unaided.
