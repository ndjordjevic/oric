Learn SystemVerilog · Lesson 0003

localparam, string concatenation & boolean operators

Source: annotated/Oric.sv, lines 63–132 (★ OSD MENU tail + ★ OSD SETTINGS DECODE). Prior: Lesson 2.

The win for today

You'll recognize localparam as a compile-time constant, see how { } concatenation builds one long string out of many short ones, and read the four operators that turn raw status bits into named booleans: ~, ==, >=, and &&-free plain comparisons.

1. localparam — a named constant, fixed forever

64localparam CONF_STR = {
65	"Oric;SS3E000000:200000;",
66	"FS1,TAP,Load TAP file;",
67	...
119	"V,v",`BUILD_DATE
120};

localparam declares a constant scoped to this module — its value is fixed at compile time and can never change while the hardware runs, unlike a wire or reg. Here the constant is CONF_STR, the configuration string the MiSTer framework parses to build this core's on-screen menu. The ... at line 67 stands for lines 67–118 — roughly fifty more quoted menu entries of exactly the same shape as lines 65–66, elided here because they add no new syntax.

Directly equivalent to a top-level const in JS or a module-level constant in Python — except here the compiler can use it to size other things (widths, loop bounds) at build time, not just as a convenient name for a fixed value.

Notice { "a", "b", "c" } here is the same concatenation operator from Lesson 1 — it glues multiple string literals end-to-end into one long string, which the MiSTer framework later parses to build the on-screen menu. `BUILD_DATE (backtick) is a preprocessor macro, defined by the `include "build_id.v" just above it — textual substitution again, same idea as Lesson 1's `include.

You don't need to learn the mini-language inside the string (the P1O[6:5],FDD Controller,... syntax) — that's the MiSTer OSD menu format, not SystemVerilog. Only the concatenation and macro-substitution around it are language syntax.

2. Turning status bits into named signals

123wire [1:0] tapeVolume = status[51:50];
124wire       tapeUseADC = status[52];
125wire       tapeRewind = status[53];
126wire [1:0] joystick_adapter = status[55:54];
127wire       tap_autorun_en = ~status[57]; // menu shows On (default) / Off
128wire [1:0] tape_load_mode = status[59:58];
129wire       tape_mode_fast = (tape_load_mode == 2'd0);
130wire       tape_mode_ultra = (tape_load_mode == 2'd1);
131wire       tape_mode_off = (tape_load_mode >= 2'd2);
132wire named_cload_rewind_en = ~status[60]; // menu shows On (default) / Off

Four operators do all the work here:

OperatorMeaningFamiliar equivalent
~bitwise NOT — flips every bit~x ≈ JS's !x for a single bit
==equality comparison, produces 1-bit true/falsesame as == in C/JS
>=numeric "greater than or equal", treats the bits as an unsigned numbersame as C/JS
(no width on wire)a plain wire foo = ...; with no [n:0] is exactly 1 bit widea boolean
Line 127 reads exactly like const tapAutorunEn = !status.bit(57); — a named boolean, computed once as a live formula, with a comment explaining the polarity (why "on" is bit value 0, not 1) because that inversion isn't obvious from the name alone.

Walking the whole block: lines 123–126 are plain slices with no operator at all — tapeVolume (2-bit tape volume), tapeUseADC, tapeRewind, and joystick_adapter (2-bit adapter choice) just pull their bits straight out of status. Line 132's named_cload_rewind_en repeats line 127's ~-inversion trick, polarity comment and all. Line 128 declares tape_load_mode, the 2-bit slice that lines 129–131 then decode into three one-bit mode flags: tape_mode_fast (== value 2'd0), tape_mode_ultra (== value 2'd1), and tape_mode_off (>= value 2'd2, i.e. either of the two remaining values) — the 2'd0/2'd1/2'd2 literals use Lesson 2's <width>'<base><value> sizing form.

~ (bitwise NOT, works bit-by-bit across a whole bus) and ! (logical NOT, treats the whole value as one true/false) look similar but differ once you're not dealing with single bits — you saw !ar in Lesson 2 used exactly because "is this whole 2-bit value zero?" is a logical question, while ~status[57] is a single-bit flip.

Quick check

Q1. What does a bare wire foo = a == b; declaration's width have to be, and why?
Answer: 1 bit — comparison operators like == always produce a single true/false result, regardless of how wide a and b are.
Q2. Why does line 127 need a comment while line 123 doesn't?
Answer: line 127 inverts the bit with ~, so the signal's active meaning ("on") doesn't match the raw stored bit value — worth flagging; line 123 is a direct, unambiguous slice with no inversion to explain.
Q3. Is localparam allowed to change value while the FPGA is running?
Answer: no — it's fixed at compile time, permanently, for the lifetime of that build.
You now have this in hand: localparam as a compile-time constant, string concatenation for building one literal from many, macro substitution via `NAME, and the operator quartet ~ / == / >= / bare-boolean wire for decoding status bits.
