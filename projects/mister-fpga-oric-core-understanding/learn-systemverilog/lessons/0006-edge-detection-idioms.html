Learn SystemVerilog · Lesson 0006

Edge-detection idioms

Source: core/Oric.sv, lines 258–309 (★ DISK IMAGE MANAGEMENT + ★ KEYBOARD & TAPE AUTORUN). Prior: Lesson 5.

The win for today

You'll recognize the two most common "did something just change?" patterns in this codebase — the delayed-copy comparison and the XOR toggle-detector — which appear dozens of times across the whole core.

1. The delayed-copy pattern: "was low, now high"

258reg    [3:0] img_mountedD;
261always @(posedge clk_sys)
262begin
263	img_mountedD <= img_mounted;
264	if(~|img_mountedD && |img_mounted) begin

img_mountedD (the trailing D means "Delayed") is updated every clock cycle to hold last cycle's value of img_mounted — because of the non-blocking-assignment rule from Lesson 4, line 263's read of img_mounted happens before the update lands, so on the very next line, img_mountedD still holds the previous cycle's value while img_mounted holds the current one.

|img_mounted is reduction-OR: true if any bit is 1. So ~|img_mountedD && |img_mounted reads as "none of the drives were mounted last cycle, AND at least one is mounted now" — i.e. exactly the clock cycle a disk image first appears.

This is the hardware version of comparing prevValue to currentValue across two ticks of a loop — like tracking wasVisible vs isVisible across animation frames to detect the exact frame something became visible.
Reduction-OR (|x, one operand) is easy to visually confuse with bitwise-OR (a | b, two operands) — same symbol, different arity and meaning. You already saw both: Lesson 4 used reduction-AND (&clr_addr); line 264 mixes reduction-OR (|img_mounted) with logical-AND (&&) combining two boolean results.

2. The XOR toggle-detector: "did this bit flip?"

302reg old_keystb = 0;
304always @(posedge clk_sys) begin
305	old_keystb <= ps2_key[10];
307end
308assign hps_key_strobe = old_keystb ^ ps2_key[10];

The keyboard interface toggles bit 10 of ps2_key every time a new key event arrives (rather than pulsing it briefly) — a common trick for crossing between different rates of activity. To turn a toggle into a clean one-cycle pulse, this compares the current bit to a one-cycle-delayed copy using XOR (^): XOR is true exactly when its two inputs differ.

^ here does the same job as writing current !== previous in JS for two booleans — "true for exactly one cycle when the value changed," which is precisely what a toggle-to-pulse converter needs.
OperatorArityMeaning
|xreduction (1 operand)true if any bit of x is 1
a | bbitwise (2 operands)OR each corresponding bit pair
a ^ bbitwise/logical XOR (2 operands)true where bits differ

Quick check

Q1. Why does the delayed-copy pattern need a separate register (img_mountedD) instead of just comparing img_mounted to itself?
Answer: comparing a signal to itself is always equal — detecting a change requires remembering the value from a previous clock cycle, which is exactly what the delayed register stores.
Q2. If ps2_key[10] toggles (flips) exactly once, for how many clock cycles will hps_key_strobe read as 1?
Answer: exactly one cycle — the cycle right after the flip, before old_keystb catches up to match the new value again.
You now have this in hand: the delayed-copy "just became true" pattern (reduction-OR + a one-cycle-old register), and the XOR toggle-to-pulse pattern — the two edge-detection idioms this codebase reuses everywhere.
