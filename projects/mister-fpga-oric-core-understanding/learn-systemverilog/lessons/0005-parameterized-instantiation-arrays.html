Learn SystemVerilog · Lesson 0005

Parameterized instantiation & unpacked arrays

Source: annotated/Oric.sv, lines 140–148 (★ CLOCK GENERATION) and 178–252 (★ HPS FRAMEWORK INTERFACE). Prior: Lesson 4.

The win for today

You'll read a module instantiation that passes compile-time parameters (not just port connections), understand what an empty port connection really means, and recognize an array-of-buses declaration.

1. A plain instantiation with one intentionally-empty port

140pll pll
141(
142	.refclk(CLK_50M),
143	.rst(0),
144	.outclk_0(clk_sys),
145	.outclk_1(CLK_VIDEO),
146	.outclk_2(),
147	.locked(locked)
148);

This is the same instantiation pattern from Lesson 2 — nothing new in the shape. Two details worth calling out precisely: .rst(0) feeds a constant literal directly into a port (perfectly legal — an input port can be tied to a fixed value, not just a named wire), and .outclk_2() leaves an output completely unconnected because this design only needs two of the PLL's three output clocks.

Passing a literal into a port is like calling a function with a hardcoded argument: makePLL({ rst: 0, outclk_2: undefined }) — you're allowed to not care about a return value (outclk_2) the same way you'd ignore an object's field you don't need.

2. Compile-time parameters: #( .PARAM(value) )

216hps_io #(.CONF_STR(CONF_STR), .VDNUM(4)) hps_io
217(
218	.clk_sys(clk_sys),
219	.HPS_BUS(HPS_BUS),
220	...

The #( ... ) block right after the module name — before the instance name and the port list — passes parameters: compile-time configuration values, distinct from the ports (which carry live signals at runtime). Here, hps_io is told its menu string (CONF_STR, the constant from Lesson 3) and that it should support 4 virtual disk drives (VDNUM(4)).

Think of a parameter as a generic/template argument — like TypeScript's Array<number> or a C++ template parameter — it configures the shape
Don't confuse #( .PARAM(value), ... ) (compile-time, comes right after the module name) with the port list ( .port(signal), ... ) (runtime, comes after the instance name). Same .name(value) visual syntax, completely different meaning and timing.

3. Arrays of buses: wire [31:0] sd_lba[4];

187wire  [31:0] sd_lba[4];
193wire   [7:0] sd_buff_din[4];

Two sets of brackets, two different jobs. [31:0] (right after wire) is the bit-width of each element — 32 bits per entry. [4] (right after the name) says there are 4 separate 32-bit buses, indexed sd_lba[0] through sd_lba[3] — one per virtual disk drive.

This is exactly a fixed-size array of same-width values: const sd_lba: number[4] in a hardware-flavored TypeScript, or Python's sd_lba = [0,0,0,0] where each entry happens to be a 32-bit integer. You saw this used later: .sd_lba_fd0(sd_lba[0]) picks out drive 0's bus specifically.

Quick check

Q1. In hps_io #(.CONF_STR(CONF_STR), .VDNUM(4)) hps_io (...), which part configures the module at compile time, and which part wires it up at runtime?
Answer: the #(.CONF_STR(...), .VDNUM(4)) block is compile-time configuration; the ( .clk_sys(clk_sys), ... ) block afterward is the runtime port/signal wiring.
Q2. What's the difference in meaning between wire [31:0] sd_lba[4] and a hypothetical wire [3:0] sd_lba?
Answer: the first is 4 separate 32-bit buses (an array); the second would be a single bus that's only 4 bits wide — completely different shape despite superficially similar-looking brackets.
You now have this in hand: tying a literal directly into a port, leaving a port deliberately unconnected, compile-time parameters via #(...) versus runtime ports, and array-of-buses declarations with two independent bracket dimensions.
