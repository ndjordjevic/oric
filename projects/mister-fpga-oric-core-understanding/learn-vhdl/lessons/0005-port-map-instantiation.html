Learn VHDL · Lesson 0005

PORT MAP instantiation — two spellings, one meaning

Source: annotated/rtl/oricatmos.vhd, lines 441–460 (★ SECTION 7 — CPU), lines 503–537 (★ SECTION 9 — ROMs), and lines 641–671 (★ SECTION 12 — PSG). Prior: Lesson 4.

The win for today

You'll read both instantiation styles this file uses — ENTITY work.X (for VHDL modules) and plain component-name instantiation (for the three COMPONENT-declared SystemVerilog modules from Lesson 3) — and recognize => as VHDL's version of SystemVerilog's .port(signal).

1. ENTITY work.X PORT MAP ( port => signal, ... );

441inst_cpu : ENTITY work.T65
442	PORT MAP(
443		Mode => "00",
444		Res_n => cont_RESETn,
445		Enable => ENA_1MHZ,
446		Clk => CLK_IN,
447		Rdy => NOT (cpu_halt OR save_halted_i),
456		Sync => cpu_sync,
457		Regs => cpu_regs_q,
460	);

inst_cpu is the instance's own label (chosen by the author, must be unique in this architecture) — like the instance name in a SystemVerilog instantiation. : ENTITY work.T65 says "this instance is one copy of the T65 entity, found in the work library" (work is the default local-project library — think of it as "look in this same codebase," not an external package). PORT MAP ( port => signal, ... ) then connects each of T65's ports to a signal here, using => where SystemVerilog uses .port(signal).

Directly: T65 inst_cpu ( .Mode("00"), .Res_n(cont_RESETn), .Clk(CLK_IN), ... ); in SystemVerilog. Same idea (named-association port connection), => replaces .name(...) syntax, and the instance label comes before the module name (label : ENTITY ...) instead of after it.

Line 447 shows a full boolean expression fed directly into a port, same as SV Lesson 7's inline-ternary-in-a-port-connection — nothing new there beyond VHDL's NOT/OR spelling.

2. Repeated for every VHDL sub-module — same shape, every time

503inst_rom0 : ENTITY work.BASIC11A -- Oric Atmos ROM
504	PORT MAP(
505		clk => CLK_IN,
506		addr => cpu_ad(13 DOWNTO 0),
507		data => ROM_ATMOS_DO
508	);

Each ROM (BASIC11A, BASIC10, PRAVETZ8D, etc.) is instantiated identically: clock in, a 14-bit address slice in, one data byte out. These giant lookup tables are literally the ROM contents compiled into block RAM — no logic to understand here beyond recognizing the now-familiar instantiation shape.

3. The other spelling: plain component-name instantiation

641psg_a: psg
642port map (
643	clock       => CLK_IN,
644	ce          => ENA_1MHZ,
645	reset       => RESETn AND KEYB_RESETn,
...
655	ioad        => "ZZZZZZZZ",
660	sel         => '1',
671	);

Here there's no ENTITY work. prefix — just label : component_name. This is the older-style instantiation VHDL uses when a COMPONENT was forward-declared (Lesson 3) rather than resolved directly against an entity. Functionally identical to the ENTITY work.X form — it's a syntax choice tied to whether the module needed a forward declaration, not a difference in what gets built.

Line 655's "ZZZZZZZZ" is eight repeated Z characters inside a double-quoted vector literal — VHDL's tri-state/high-impedance value (matching SystemVerilog's 'Z from SV Lesson 1), applied to every bit of an 8-bit port at once. Same electrical meaning as SV's 'Z, just written per-bit inside a string-like literal instead of as a single width-filling token.

Quick check

Q1. What does => connect, on which sides, in a PORT MAP?
Answer: the module's port name on the left, the signal/expression from this architecture connecting to it on the right — same relationship as SystemVerilog's .port(signal).
Q2. Is there a functional difference between inst_cpu : ENTITY work.T65 PORT MAP(...) and psg_a: psg port map(...), beyond syntax?
Answer: no — both instantiate one copy of a module and wire its ports; the difference is purely which VHDL instantiation style is used (direct entity reference vs. a previously-declared COMPONENT), driven by whether the target module needed a forward declaration.
You now have this in hand: both VHDL instantiation styles (ENTITY work.X and plain component-name), the label : module PORT MAP ( port => signal ) shape as VHDL's counterpart to SV's named-port instantiation, and the "ZZ...Z" multi-bit tri-state literal.
