Learn VHDL · Lesson 0006

PROCESS & rising_edge

Source: annotated/rtl/oricatmos.vhd, lines 466–476 (p_save_halt) and lines 859–873 (★ SECTION 18 — Pravetz bank/shadow register). Prior: Lesson 5. This is VHDL's counterpart to SystemVerilog Lesson 4 — arguably the most important lesson in this series.

The win for today

You'll read a clocked PROCESS block — VHDL's equivalent of SystemVerilog's always @(posedge clk) — and see that IF/ELSIF/END IF plays the same priority role as SV's if/else if.

1. PROCESS (sensitivity list) BEGIN ... END PROCESS;

466p_save_halt : PROCESS (CLK_IN)
467BEGIN
468	IF rising_edge(CLK_IN) THEN
469		IF save_halt = '0' THEN
470			save_halted_i <= '0';
471		ELSIF cpu_sync = '1' AND ENA_1MHZ = '0' THEN
472			save_halted_i <= '1';
473		END IF;
474	END IF;
475END PROCESS;

p_save_halt : PROCESS (CLK_IN) BEGIN ... END PROCESS; declares a block of sequential logic — statements execute top-to-bottom in the order written, unlike the concurrent WHEN/ELSE assignments from Lesson 4. (CLK_IN) is the sensitivity list: this process re-runs whenever CLK_IN changes (rising or falling edge — the IF rising_edge(CLK_IN) THEN line inside then filters that down to "only actually do anything on the rising edge").

SystemVerilog folds the edge-sensitivity directly into the trigger: always @(posedge CLK_IN) begin ... end already means "only on the rising edge," so there's no separate if rising_edge(...) check needed inside. VHDL's older style instead lists the whole signal in the sensitivity list, then explicitly tests rising_edge(...) as the first line inside — same end result, one extra layer of ceremony.

<= inside this PROCESS is the clocked, non-blocking form (the second meaning from Lesson 4's gotcha) — behaving exactly like SystemVerilog's <= inside always @(posedge clk): reads use old values, all updates land together at the next clock edge.

Two more details in this excerpt. First, p_save_halt : is a process label — optional (the second example below has none), purely a name for humans and tools, like an instance label. Second, the signals: save_halt is the save-state engine's "please stall the CPU" request, and save_halted_i is the "CPU is now stalled" flag this process feeds back (it's one of the two stall sources gating the CPU's Rdy port in Lesson 5). The ELSIF condition — cpu_sync = '1' AND ENA_1MHZ = '0' — waits until T65 is parked exactly at an opcode-fetch boundary (and not on the very tick that completes the fetch) before latching the stall, so a snapshot captures a clean between-instructions CPU state.

2. IF / ELSIF / ELSE / END IF; — the sequential if-chain

IF cond1 THEN ... ELSIF cond2 THEN ... ELSE ... END IF; is VHDL's if/else-if/else ladder, evaluated top to bottom, first match wins — direct counterpart to SystemVerilog's if (cond1) ... else if (cond2) ... else .... Every IF must be closed with its own END IF; — unlike SV, which needs no closing keyword for a single-statement or begin/end-wrapped if.

ELSIF is one word, no space, no second E — a very common typo target coming from other languages' else if (two words) or SV's identical two-word form. VHDL specifically wants ELSIF.

3. A second example: reset-or-write priority

859PROCESS (CLK_IN)
860BEGIN
861	IF rising_edge(CLK_IN) THEN
862		IF RESETn = '0' OR pravetz_mode = '0' THEN
863			pravetz_bank <= '0';
864			pravetz_shadow <= '0';
865		ELSIF ula_phi2 = '1'
866		      AND cpu_rw = '0'
867		      AND cpu_ad(15 DOWNTO 8) = X"03"
868		      AND cpu_ad(7 DOWNTO 2) = "100000" THEN
869			pravetz_bank <= cpu_ad(1);
870			pravetz_shadow <= cpu_ad(0);
871		END IF;
872	END IF;
873END PROCESS;

Same shape as SystemVerilog Lesson 4's RESET & RAM CLEAR block: reset condition takes priority (clears two registers), otherwise a specific CPU-write condition (a multi-line AND chain, exactly like SV's multi-line boolean conditions) updates the same two registers from two individual address bits. No begin/end needed for the multi-statement branches — VHDL's THEN ... ELSIF structure already delimits each branch without extra block syntax.

Decoding that write condition piece by piece: ula_phi2 = '1' means it's the CPU's half of the clock cycle (as opposed to the ULA's video-fetch half), cpu_rw = '0' means the CPU is writing, and the two address comparisons — high byte equal to X"03", then bits 7 down to 2 equal to binary "100000" — together select the four addresses $0380–$0383. The two remaining low address bits then become the data: pravetz_bank <= cpu_ad(1) and pravetz_shadow <= cpu_ad(0) — writing to any of those four addresses sets the Pravetz FDC-ROM bank flag and the shadow-ROM flag straight from address bits 1 and 0.

Quick check

Q1. What's the VHDL counterpart to SystemVerilog's always @(posedge clk) begin ... end, spelled out fully?
Answer: PROCESS (clk) BEGIN IF rising_edge(clk) THEN ... END IF; END PROCESS; — the sensitivity list plus an explicit rising-edge check, versus SV's single combined trigger.
Q2. Inside a PROCESS, does <= behave like VHDL's continuous assignment from Lesson 4, or like SystemVerilog's non-blocking assignment?
Answer: like SystemVerilog's non-blocking assignment — reads see old values, updates land together at the next clock edge; it's the same symbol as continuous assignment, but its meaning switches once you're inside a PROCESS.
You now have this in hand: PROCESS (sensitivity-list) as the clocked-block trigger, the explicit rising_edge(...) test VHDL needs (that SV bakes into @(posedge ...)), IF/ELSIF/END IF as the sequential priority ladder, and confirmation that <= switches meaning depending on whether you're inside a PROCESS.
