COMPONENT forward declarations

Source: annotated/rtl/oricatmos.vhd, lines 322–387 (★ SECTION 5 — Component declarations). Prior: Lesson 2.

The win for today

You'll recognize COMPONENT ... PORT (...) END COMPONENT; as a forward declaration — telling VHDL about another module's interface before instantiating it — and see why it's needed here specifically for the Verilog sub-modules.

1. The shape: identical to an entity's port list, wrapped differently

326COMPONENT keyboard
327	PORT (
328		clk_sys : IN STD_LOGIC;
329		reset : IN STD_LOGIC;
330		key_pressed : IN STD_LOGIC;
...
339		swrst : OUT STD_LOGIC
340	);
341END COMPONENT;

Compare this to Lesson 1's ENTITY oricatmos IS PORT (...) END; — same shape (a name, a port list with direction/type per pin), wrapped in COMPONENT ... END COMPONENT; instead of ENTITY ... END;. This block doesn't *define* the keyboard's behavior — it just tells this file "there exists a module called keyboard with these exact pins," so the compiler can check instantiations of it later in this file for correctness.

This is exactly a C function prototype (int foo(int, char*); in a header, with the real implementation in a separate .c file) or a TypeScript interface (an ambient declaration) — describing a shape without providing the body.
SystemVerilog doesn't need this step at all — you can instantiate any module directly by name (as you did throughout Oric.sv, and as this same file does for its VHDL sub-modules via ENTITY work.T65, coming in Lesson 5) as long as the compiler can find its source. Here the COMPONENT-declared modules — keyboard, joystick, and psg — happen to be exactly the three written in SystemVerilog/Verilog (keyboard.sv, joystick.sv, psg.v). Direct ENTITY work.X instantiation of a foreign-language module is tool-dependent (not guaranteed by the VHDL standard), so the classic, most-portable way to instantiate across the language boundary is via a matching-named COMPONENT declaration — which is the style the author chose for these three.

2. Why three components, and why these three specifically

keyboard, joystick, and psg are declared here because they're the only sub-modules in this file written in a *different* HDL (SystemVerilog/Verilog) than the VHDL modules instantiated elsewhere (which use the more modern ENTITY work.X style you'll see in Lesson 5 — no forward declaration needed for same-language, same-library modules).

Don't spend time trying to learn keyboard.sv, joystick.sv, or psg.v's internals here — from this VHDL file's point of view they're opaque boxes with a fixed pin interface. Their actual SystemVerilog syntax is in scope of the sibling ../learn-systemverilog/ workspace, not this one, per this workspace's MISSION.md.

Quick check

Q1. What's the practical difference between a COMPONENT declaration and the ENTITY it's forward-declaring?
Answer: none in terms of pin shape — a COMPONENT declaration just repeats the entity's port list, without any implementation, so this file can check instantiations against it before that module's actual body is compiled/linked in.
Q2. Why do only keyboard, joystick, and psg get a COMPONENT declaration, while ROMs and the CPU (T65) don't?
Answer: those three are the sub-modules written in SystemVerilog/Verilog rather than VHDL; instantiating a foreign-language module directly with ENTITY work.X is tool-dependent, so the portable convention is a matching-named COMPONENT declaration — while the VHDL modules can safely use ENTITY work.X directly, needing no forward declaration.
You now have this in hand: reading COMPONENT ... END COMPONENT; as a forward declaration (like a function prototype), and understanding specifically why this file needs it for its three non-VHDL sub-modules.
