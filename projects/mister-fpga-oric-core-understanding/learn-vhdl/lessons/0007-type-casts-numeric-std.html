Learn VHDL · Lesson 0007

Type casts & numeric_std

Source: annotated/rtl/oricatmos.vhd, lines 399–406 (Pravetz window decode) and lines 478–493 (★ SECTION 8 — RAM bus arbitration). Prior: Lesson 6.

The win for today

You'll read VHDL's strict-typing casts (unsigned(...), TO_UNSIGNED(...), STD_LOGIC_VECTOR(...)) and understand why VHDL needs them where SystemVerilog just lets you compare numbers directly.

1. Why casts exist: VHDL's stricter type system

401unsigned(cpu_ad(7 DOWNTO 0)) >= TO_UNSIGNED(16#10#, 8)

(Line 401 is one clause of the pravetz_extension_window decode, lines 399–402 — the WHEN chain Lesson 4 sampled a different clause of.) cpu_ad(7 DOWNTO 0) is a STD_LOGIC_VECTOR — just a bundle of individual bits with no built-in idea of "numeric value." To do a numeric comparison (>=) you must first tell VHDL to treat that bit-bundle as a number: unsigned(...) casts it to the unsigned type (from the numeric_std package imported in Lesson 1). TO_UNSIGNED(16#10#, 8) builds the other side of the comparison directly as an 8-bit unsigned number, from the value 16#10# (VHDL's hex-literal syntax: base#value# — here, hex 10, i.e. decimal 16).

SystemVerilog collapses this entire distinction — a wire [7:0] can be compared numerically with >= directly, no cast needed, because SV doesn't separate "bit vector" from "number" as strictly. VHDL's extra ceremony here (unsigned(...), TO_UNSIGNED(...)) exists specifically because STD_LOGIC_VECTOR deliberately carries no numeric interpretation on its own — it's a design choice for type safety, not an oversight.
16#10# is not the same as VHDL's other hex literal you already know, X"10". Both mean hex 0x10 (decimal 16), but base#value# is a general based-literal notation usable directly as a plain number (as the first argument to TO_UNSIGNED here, which expects a number, not a bit vector), while X"..." directly produces a STD_LOGIC_VECTOR-typed literal.

2. Casting back: STD_LOGIC_VECTOR(TO_UNSIGNED(...))

You'll see this combined pattern later in the file (Section 17): STD_LOGIC_VECTOR(TO_UNSIGNED(16#2735#, 14)) — build a number with TO_UNSIGNED, then cast the result back to a plain bit vector with STD_LOGIC_VECTOR(...) so it can be compared against cpu_ad(13 DOWNTO 0) (which is itself a STD_LOGIC_VECTOR, not an unsigned).

Think of STD_LOGIC_VECTOR, unsigned, and signed as three different VHDL types that all happen to store the same bits — like TypeScript's structural types where the same underlying data needs an explicit cast/assertion to be treated as a different named type before certain operations (arithmetic, numeric comparison) are allowed on it.

3. Bus arbitration: reading a mux with no new syntax

483ram_ad <= ula_AD_SRAM WHEN (ula_PHI2 = '0') ELSE
484	cpu_ad(15 DOWNTO 0);
485ram_d <= cpu_do;
486SRAM_DO <= ram_q;
487ram_cs <= '0' WHEN RESETn = '0' ELSE
488	ula_CE_SRAM;
489ram_oe <= '0' WHEN RESETn = '0' ELSE
490	ula_OE_SRAM;
491ram_we <= '0' WHEN RESETn = '0' ELSE
492	ula_WE_SRAM;
493phi2 <= ula_PHI2;

Nothing new here — this is Lesson 4's WHEN/ELSE pattern, applied to the RAM interface. Lines 483–484: when the ULA's PHI2 clock-half is low (the video-fetch half-cycle), the RAM address ram_ad comes from the ULA's own video-scan address ula_AD_SRAM; otherwise it comes from the low 16 bits of the CPU's address bus. Lines 485–486 are the simplest continuous assignments possible (Lesson 4's bare form): RAM write data ram_d always comes from the CPU's data-out cpu_do, and the RAM's read data ram_q is exported on the SRAM_DO output port. Lines 487–492 are three copies of one idea: the RAM's chip-select (ram_cs), output-enable (ram_oe), and write-enable (ram_we) strobes are all forced to '0' (inactive) while RESETn is asserted, and otherwise follow the ULA's corresponding control outputs ula_CE_SRAM / ula_OE_SRAM / ula_WE_SRAM — the ULA, not the CPU, runs the RAM timing. Line 493 exports the clock phase itself on the phi2 output port — which, full circle, is exactly the port Oric.sv left unconnected with .phi2 () in SV Lesson 7.

Quick check

Q1. Why can't cpu_ad(7 DOWNTO 0) >= TO_UNSIGNED(16#10#, 8) be written without the unsigned(...) cast around cpu_ad(7 DOWNTO 0)?
Answer: because cpu_ad(7 DOWNTO 0) is a STD_LOGIC_VECTOR, which has no built-in numeric comparison operators — VHDL requires it to be explicitly cast to unsigned before >= can treat it as a number.
Q2. What's the difference between X"10" and 16#10#?
Answer: both represent hex 0x10 (decimal 16), but X"10" produces a STD_LOGIC_VECTOR-typed bit-vector literal directly, while 16#10# is a general based-number literal usable as a plain numeric value, e.g. as an argument to TO_UNSIGNED.
You now have this in hand: why VHDL needs explicit casts (unsigned(...), TO_UNSIGNED(...), STD_LOGIC_VECTOR(...)) where SystemVerilog doesn't, the base#value# literal notation, and confirmation that the RAM arbitration logic is just Lesson 4's WHEN/ELSE idiom again.
