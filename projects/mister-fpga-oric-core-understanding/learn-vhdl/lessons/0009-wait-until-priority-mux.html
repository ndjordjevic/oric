WAIT UNTIL & the priority-mux process

Source: annotated/rtl/oricatmos.vhd, lines 876–927 (★ SECTION 19 — CPU data-in mux). Prior: Lesson 8.

The win for today

You'll read a third way VHDL expresses "on the clock edge" — PROCESS BEGIN WAIT UNTIL rising_edge(clk); ... END PROCESS; with no sensitivity list at all — and read a long IF/ELSIF chain as a hardware priority mux, mirroring SystemVerilog Lesson 8's chained ternary.

1. A PROCESS with no sensitivity list, using WAIT UNTIL instead

882PROCESS BEGIN
883
884	WAIT UNTIL rising_edge(clk_in);
885
886	IF cpu_rw = '1' AND ula_phi2 = '1' AND patch_active = '1'
887	      AND ula_CSROMn = '0' AND cont_MAPn = '1' AND cont_ROMDISn = '1' THEN
888		cpu_di <= patch_data;
889	ELSIF ...

Compare to Lesson 6's PROCESS (CLK_IN) BEGIN IF rising_edge(CLK_IN) THEN ... END IF; END PROCESS;. This is a third equivalent style: PROCESS with an empty sensitivity list () (or none at all, as here), where WAIT UNTIL rising_edge(clk_in); as the very first statement does the same job — pause here until the next rising edge, then run everything below once, then loop back to the WAIT UNTIL and repeat forever.

All three forms — always @(posedge clk) (SV), PROCESS(clk) ... IF rising_edge(clk) THEN (Lesson 6), and PROCESS BEGIN WAIT UNTIL rising_edge(clk); ... (here) — describe the exact same hardware: a bank of flip-flops sampling their inputs once per clock rising edge. Different eras/authors of VHDL style prefer different spellings; recognizing all three as "the same clocked-block idea" matters more than picking a favorite.
A WAIT UNTIL statement can only appear inside a PROCESS with no sensitivity list — mixing a sensitivity list AND a WAIT statement in the same process is illegal. That's why line 882 has bare PROCESS BEGIN, no (CLK_IN) after it, unlike Lesson 6's examples.

2. Reading the long IF/ELSIF chain as a priority mux

This whole process is one big decision: which of ~9 sources gets to drive cpu_di (the CPU's data-in bus) this cycle. Reading it top to bottom, first match wins — the exact same priority-list idea as SystemVerilog Lesson 8's chained ternary, and VHDL Lesson 4's chained WHEN/ELSE, just written as a sequential IF/ELSIF ladder inside a PROCESS instead of a concurrent expression.

Notice there's no final unconditional ELSE here — if none of the IF/ELSIF conditions match, cpu_di simply isn't assigned this cycle, meaning it holds its previous value (VHDL's rule: an unassigned signal in a clocked process retains its old value, exactly like an unassigned reg in a SystemVerilog always block). This is different from SV Lesson 8's chained ternary, which always has a defined fallback because a continuous wire assignment can never be "left unassigned."

Each condition itself is nothing new — a chain of AND-combined comparisons, same as Section 6 and Section 18. The only genuinely new idea in this whole lesson is the sensitivity-list-free PROCESS/WAIT UNTIL form.

Quick check

Q1. Can a PROCESS have both a sensitivity list like (CLK_IN) and a WAIT UNTIL statement inside it?
Answer: no — a process with a sensitivity list cannot contain a WAIT statement, and vice versa; they're two mutually exclusive ways of expressing the same "wake up on this event" idea.
Q2. If none of the IF/ELSIF conditions in this process are true on a given clock edge, what happens to cpu_di?
Answer: it keeps its previous value — an unassigned signal in a clocked process implicitly holds state, the same behavior as an unassigned reg in a SystemVerilog always block.
You now have this in hand: the sensitivity-list-free PROCESS BEGIN WAIT UNTIL rising_edge(clk); ... END PROCESS; form as a third spelling of "clocked block," and reading a long IF/ELSIF chain as a hardware priority mux with implicit hold-state on no match.
