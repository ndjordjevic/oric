Learn VHDL · Lesson 0004

Concurrent assignment & WHEN/ELSE

Source: annotated/rtl/oricatmos.vhd, lines 395–434 (★ SECTION 6 — Reset, Pravetz mode decode & controller mux). Prior: Lesson 3. This is VHDL's counterpart to SystemVerilog Lessons 1, 3, and 8 combined.

The win for today

You'll read VHDL's <= continuous assignment and its WHEN ... ELSE priority-chain idiom — the single most common pattern in this file, doing the same job as SystemVerilog's chained ternary.

1. <= is VHDL's assign, not its non-blocking clocked assignment

397RESETn <= (NOT RESET AND KEYB_RESETn);
This is the single sharpest trap moving from SystemVerilog to VHDL: <= means completely different things depending on context. Outside a PROCESS (as here, directly in the architecture body after BEGIN), <= is a continuous assignment — exactly SystemVerilog's assign, permanently live, re-evaluating the instant any input changes. Inside a PROCESS (VHDL Lesson 6), the very same <= symbol becomes VHDL's non-blocking assignment, matching SystemVerilog's <= inside an always block. Same symbol, different meaning, determined entirely by whether you're inside a PROCESS or not.
Line 397 is exactly assign RESETn = ~RESET & KEYB_RESETn; in SystemVerilog — NOT/AND/OR are VHDL's spelled-out boolean operators, doing the same job as ~/&/|.

The three signals: RESET is the entity's external reset input (Lesson 1), KEYB_RESETn is the keyboard's reset request — the inverted swrst output of the keyboard module (Lesson 3), already active-low as its trailing n indicates — and RESETn is the combined internal reset the rest of the file uses: it goes low (active) when either the external reset is asserted or the keyboard's reset combo is pressed.

2. WHEN ... ELSE — VHDL's ternary, chainable

398pravetz_mode <= '1' WHEN rom = "10" ELSE '0';

Read this right-to-left-ish: "pravetz_mode becomes '1' WHEN the condition rom = "10" holds, ELSE it becomes '0'." This is VHDL's ternary — but notice the word order is reversed from SystemVerilog's cond ? a : b: VHDL puts the "then" value first, then the condition, then the "else" value.

SV: wire pravetz_mode = (rom == 2'b10) ? 1'b1 : 1'b0; — same logic, condition-first. VHDL: value WHEN condition ELSE value — value-first. Same tool, mirrored word order; worth deliberately practicing the re-ordering until it stops requiring conscious translation.

Note "10" — double-quoted, a 2-bit vector literal (Lesson 1's quoting rule) — being compared with = to the rom signal: the entity's 2-bit ROM-selection input (fed from Oric.sv's rom register, SV Lesson 8), where value "10" is the Pravetz-8D choice. So pravetz_mode is the "this machine is currently a Pravetz" flag used throughout the rest of the file.

3. Chaining WHEN/ELSE — same priority-list idiom as SV's chained ternary

408cont_MAPn <= '0' WHEN pravetz_mode = '1'
409                      AND pravetz_shadow = '1'
410                      AND cpu_ad(15 DOWNTO 14) = "11" ELSE
411             '1' WHEN pravetz_mode = '1' ELSE
412             md_MAPn;

Exactly the chained-ternary priority pattern from SystemVerilog Lesson 8 (the rom_sel example): read top-to-bottom, first matching condition wins. Line 408–410's condition uses AND to combine three sub-conditions into one (VHDL's word-based boolean operators again). If none of the explicit WHEN conditions match, the final bare value (md_MAPn, line 412 — no WHEN attached) is the unconditional fallback, just like the bare tape_addr at the end of SV Lesson 8's second example.

The names here: cont_MAPn is the (active-low) memory-MAP control line going to the ULA — the cont_ prefix marks the merged "controller" result — while md_MAPn is the Microdisc disk controller's version of the same line, and pravetz_shadow is the shadow-ROM flag set by the clocked register in Lesson 6. In plain terms: in Pravetz mode this logic computes the MAP line itself (from the shadow flag and the top two address bits); otherwise the Microdisc's output passes through untouched. Lines 408–412 are also just the first statement of a whole block of these muxes (lines 413–434) — one per expansion-port signal (cont_ROMDISn, cont_IOCONTROLn, the SD-card bus, and so on), every one following the same "pravetz_ value WHEN pravetz_mode = '1' ELSE md_ value" shape.

SV equivalent: 
wire cont_MAPn =
  (pravetz_mode && pravetz_shadow && cpu_ad[15:14]==2'b11) ? 1'b0 :
  pravetz_mode ? 1'b1 :
  md_MAPn;
Identical priority-list logic; VHDL just writes each condition after its value instead of before.

4. Bit-slicing and comparison: same ideas, VHDL spelling

400cpu_ad(15 DOWNTO 8) = X"03"

(This is one clause from the middle of lines 399–402 — the pravetz_extension_window decode, another WHEN/ELSE like the ones above; the full statement returns in Lesson 7.) cpu_ad(15 DOWNTO 8) is VHDL's bit-slice syntax — parentheses instead of SystemVerilog's square brackets, but the same "high bit to low bit, inclusive" meaning as cpu_ad[15:8]. X"03" is a hex literal (the X prefix marks hex, matching SV's 'h base marker) — here an 8-bit hex value 03, compared for equality with =.

Quick check

Q1. In x <= '1' WHEN cond ELSE '0'; written directly in the architecture body (not inside a PROCESS), is this a continuous or a clocked assignment?
Answer: continuous — outside a PROCESS, <= behaves like SystemVerilog's assign, live and re-evaluated the instant any input changes.
Q2. Rewrite y <= a WHEN c1 ELSE b WHEN c2 ELSE d; as a SystemVerilog ternary chain.
Answer: wire y = c1 ? a : c2 ? b : d; — same priority order, condition-first instead of value-first.
You now have this in hand: VHDL's dual-meaning <= (continuous outside a PROCESS, clocked inside one), the WHEN/ELSE ternary with its reversed value-first word order, chained WHEN/ELSE as a priority list, parenthesis-based bit-slicing, hex literals, and word-spelled boolean operators.
